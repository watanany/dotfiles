# グローバル開発ルール

## コミュニケーション

- 日本語で応答する。コード・コマンド・技術用語は英語のまま使用する
- 簡潔に。質問は1〜2個に絞り、Yes/Noで進められる提案をする
- 質問への回答時にファイルを変更しない。変更は明示的な指示を待つ

## 作業ディレクトリ

- `~/sanctum/projects/` — 開発プロジェクト群
- `~/Documents/` — ドキュメント・調査資料
- `~/Desktop/` — 一時作業

## コマンド

- 属人性を排除しテキストとして残すため、AWS等の操作は可能な限りCLIで行う
- `find` → `fd` / `grep` → `rg` / `git checkout` → `git switch`

## Git運用

- コード変更の前にワークツリーに未コミットの変更やuntrackedファイルがないか確認する（前回の作業の残りかもしれない）
- Conventional Commits（日本語）: `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, `test:`
- PR: `gh pr create --draft --assignee @me`

## PR Description

### 正確性（最優先）
- 変更内容・ファイル数・テスト結果・コマンドは実際のdiff/実行結果と完全一致させる
- 実態と異なる表現を使わない（例: 「リネーム」ではなく「削除+新規作成」）
- 参照PRは「〜を踏襲」ではなく具体的なPR URLを記載する

### 構成
- PRオープン = レビュー依頼。作業者が品質担保・動作確認を済ませた前提で書く
- PR単体で文脈がわかるようにする（作業計画を読んでいない人にも伝わる）
- 「レビュー時に重点確認」等、レビュアーに責任を委ねる表現は使わない
- 全体像→詳細の階層構造にする: 概要で目的がわかり、必要に応じて詳細にドリルダウンできるようにする。詳細は `<details>` で折りたたむ(折りたたみ内も構造化する)
- 複雑なPR(複数の変更目的、依存チェーン等)ではツリー形式が有効。単純なPRでは不要
- 目的が異なる変更は独立セクションで分離する
- 計画外の変更は「追加で変更した」旨を明示する
- 情報は省略しない。ただし冗長にもしない

### 一貫性
- 同種のPRは過去の同種PRと同じ形式にする。作成前に過去PRを確認する
- PR templateのセクション構成に従う
- チームのユビキタス言語に従い、表記揺れしない

### 必須項目
- PBI/タスクのURL。URLが不明な場合はユーザーに確認する
- 動作確認: 実施環境(ローカル/Staging/Production)と方法を正確に記述する
- 動作確認用の一時リソース(S3, terraform PR等)がある場合、リリース後の削除/Revert予定を明記する
- 無関係なテスト失敗がある場合、developでも同様のエラーが出る旨を簡潔に記載する(詳細は不要)
- スクリーンショットは必要最小限にする

### 表記ルール
- コード的要素(テーブル名・カラム名・ファイル名・SQL等)にはバッククォートを付与する。概念的用語(レイヤ名・サービス名等)には不要
- モデル名はフルネームで記載する(`/` による省略禁止)
- カッコは半角 `()` を使う(全角 `（）` 禁止)
- PRリンクは `#87` ではなくフルURLにする(Notion転記時にリンクが壊れるため)
- `#1`, `#2` 等の番号表記はGitHub上でPRリンクに自動変換されるため注意する
- 依存チェーンやモデル構造はツリー形式で可視化する

### プロセス
- description作成前に、まずファイル(`/tmp/pr-desc.md` 等)に書き出して確認する。確認後にPRを更新する
- 作成後にセルフレビューを行う(数値整合性 + レビュアー視点)
